## 1. Суть проекта

Персональный трекер финансов с контролем дневного лимита трат. Приложение перехватывает push-уведомления Kaspi, автоматически распознаёт суммы операций и считает, сколько можно потратить в день, чтобы дожить до следующего поступления без минуса. При превышении лимита — уведомление.

## 3. Архитектура

- **Pattern:** MVVM + репозиторий, источник данных абстрагирован (`FinanceRepository`) — локальный/облачный взаимозаменяемы.
- **Локально:** Room (SQLite) — источник правды. Деньги хранятся в **тиынах** (`Long`), без float.
- **Облако:** Supabase (PostgreSQL + Auth + Row Level Security), схема с RLS готова в `docs/supabase_schema.sql`.
- **Фон:** WorkManager для отложенной выгрузки очереди.
- **UI:** Jetpack Compose, Material 3, язык — русский.
- **Принцип данных:** сырой текст пуша сохраняется целиком — для перепарсинга при смене формата.

## 4. Как работает учёт трат

Источник — **только push-уведомления Kaspi** (не баланс карты). Цепочка: пуш → `NotificationListenerService` → парсер (regex) → Room → сумма за день вычитается из дневного лимита. Формула:

```
дневной лимит = (остаток − обязательные платежи) ÷ дней до поступления
остаток на сегодня = дневной лимит − потрачено сегодня
```

## 5. Экраны и управление записями

Два экрана, навигация — простым состоянием (`var screen by remember`), без `navigation-compose` (для двух экранов надёжнее и без лишней зависимости).

- **Главный экран — витрина.** TopAppBar с заголовком «Лимит» и кнопкой «Записи» справа, крупный остаток дневного лимита, список трат за сегодня. Карточки некликабельны — никаких интерактивных жестов.
- **Экран «Записи» — управление.** Все транзакции (через `TransactionDao.observeAll`), сгруппированы по датам (Сегодня / Вчера / дд.ММ.гггг), новые сверху. В каждой строке мерчант, сумма, тип + время и две явные кнопки:
  - **«Изменить»** → диалог: сумма (`Money.parseToTiyn`, текущее значение подставлено), тип (покупка / перевод / пополнение), мерчант (можно пустым). Валидация: сумма обязательна и > 0, иначе подсветка ошибки и блокировка сохранения. Правленые записи помечаются «изменено».
  - **«Удалить»** → подтверждение.

> Изначально правка/удаление были на карточках главного экрана через жесты (tap / long-press), но на реальном устройстве жесты не срабатывали. Однозначную причину статикой найти не удалось (вложенного скролла и оверлеев нет, состояние поднято корректно), поэтому жесты убраны, а управление вынесено на явный экран «Записи» — надёжный путь.

Технически: добавлено поле `transactions.edited_at` (null = не правилась). При любой правке `synced` сбрасывается в 0 — изменённая версия повторно уйдёт в Supabase. **Сырое уведомление (`raw_notifications`) не трогается** — остаётся эталоном для сверки с парсером. Лимит и списки пересчитываются автоматически через существующие `Flow`. Версия схемы БД поднята 1 → 2. Удаление пока «жёсткое» (`DELETE`), soft delete не внедрён.

## 6. Текущие ограничения

- **Нюанс лимита (открыт):** остаток теперь авто-списывается, а `SpendingLimitCalculator` ещё отдельно вычитает «потрачено сегодня» — сегодняшняя трата учитывается ~1+1/дней раз. Решение (считать лимит от остатка на начало дня) согласовано, но пока НЕ применено.
- **Бэкап односторонний:** выгрузка в Supabase есть, а **закачки с сервера (restore на новом устройстве) нет** — это отдельная доработка.
- **Облачный бэкап требует** ключей Supabase в `local.properties` и входа в аккаунт; без них — оффлайн.
- **Мультивалютность — только задел:** поле `currency` есть, конвертации по курсу нет; сводки по категориям считаются в KZT.
- **Форматы пушей Kaspi не сверены** на большом числе реальных операций (регулярки по типовым шаблонам).
- **Зависимость от системы:** нужен доступ к уведомлениям + работа в фоне (иначе пропуск трат).

## 7. Технические инциденты и фиксы (решены)

- `--jvm-vendor` / Daemon JVM criteria — лечилось бампом Gradle до 8.14.4 и фиксированным JDK 17 в настройках Android Studio.
- Деприкейшн `LocalLifecycleOwner` — исправлен (переход на `lifecycle-runtime-compose`).
- **Баг парсера (поймал юнит-тест):** контрагент из «…от Имя» не извлекался. Причина — `\bот` в регулярке: в Java/Kotlin regex `\w`/`\b` по умолчанию не считают кириллицу словом, граница перед русским «от» не срабатывала. Заменено на якорь `(?:^|\s)от\s+…`.
- **Неработавшие жесты на карточках** — устранены переносом управления на экран «Записи» (см. п. 5).

### К чему пришли по переводам (ключевой вывод)
- **Kaspi НЕ шлёт push на исходящие переводы**, и остатка в пуше тоже нет (виден только во внутренней ленте «Сообщения» самого Kaspi). Проверено пользователем на устройстве.
- Listener уже читает `bigText` (развёрнутый текст) — то есть берёт максимум из того, что система отдаёт. Но данных по переводу там физически нет.
- **Следствие:** автономно поймать исходящий перевод перехватом уведомлений нельзя. Пути:
  1. **Ручной ввод перевода** — кнопка «+» и форма. Просто, надёжно, не автономно. Рекомендуемый старт.
  2. **AccessibilityService** — читать экран Kaspi в момент перевода. Автономно, но хрупко (редизайн ломает) и долго — отдельный крупный этап.
- Окончательно способ ещё не зафиксирован; склоняемся к ручному вводу сейчас + Accessibility опционально позже.

### Что хотим сделать дальше — меню настроек как хаб
1. **Смена валюты.** Развилка: (А) валюта только новых операций — дёшево, но в сводках мешаются валюты; (Б) базовая валюта с пересчётом по курсу — нужен для переезда KZ↔KG. План: UI выбора базовой валюты сейчас, подгрузка/кэш курса (бесплатный API) + конвертация — следующим шагом.

3. **Подкарманы (конверты/копилки).** Виртуальные кошельки внутри остатка: откладываешь N под цель. ## 12. Сессия 4 задач (12.06.2026) — soft delete, ручной ввод, автосписание, Supabase

Выполнялись строго по порядку, без фич сверх описанного. Сборку проверял пользователь в Android Studio (CLI-сборки нет: `gradlew`/`gradle` недоступны).

### Задача 1 — Soft delete ✅ (собрано, ОК)
`transactions.deleted_at`, схема **v3→v4** (`MIGRATION_3_4`). `deleteById` → UPDATE (deleted_at, synced=0, edited_at). `deleted_at IS NULL` во всех пользовательских выборках; `unsynced` БЕЗ фильтра (удаления должны синкаться). `supabase_schema.sql` += deleted_at.
**Решения сам:** имя `deleteById` сохранил (+параметр `now`); `unsynced` намеренно без фильтра.

### Задача 2 — Ручное добавление (FAB «+») ✅
`Transaction.isManual` (raw_id==null), `addManualTransaction` (raw_id=NULL, категория через Categorizer если не задана), FAB на главном и «Записях», `AddTransactionDialog` (сумма/тип/мерчант/категория/Material3 DatePicker), метка «вручную».
**Решения сам:** валюту в форму не выносил (KZT по умолчанию); дата через пикер, время = «сейчас»; FK raw_id уже nullable — миграцию не трогал.

### Задача 3 — Автосписание с остатка ✅
`SettingsRepository.applyBalanceDelta` (атомарно в DataStore), `domain/BalanceEffect` (чистая арифметика), `TransactionDao.findById`, единый путь в `FinanceRepositoryImpl` (Room-часть в `withTransaction`, сдвиг остатка после). `BalanceEffectTest` (8 кейсов). Подпись поля остатка обновлена.
**Решения сам:** остаток в DataStore (не Room) → межхранилищной атомарной транзакции нет; защита от двойного счёта — единый путь + atomic-delta + проверка `deleted_at`. Тесты — на чистом `BalanceEffect`.
**✅ ЗАКРЫТО (23.06.2026):** двойной счёт сегодняшних трат в калькуляторе исправлен. Т.к. остаток теперь живой (траты списываются интрадей), `SpendingLimitCalculator.compute` считает лимит от остатка на начало дня (`startOfDayBalance = balanceTiyn + spentTodayTiyn`) → `daily` стабилен весь день, трата вычитается ровно один раз (в `remaining`). Все 4 места вызова уже шлют живой остаток + spentToday — не менялись. Тесты: поправлен `exceeded_whenSpentOverLimit` (остаток теперь живой) + добавлен регресс `dailyLimit_stableAfterIntradaySpend`.

### Задача 4 — Supabase-синк ✅ (код; нужен бэкенд от юзера)
Плагин `kotlin-serialization` + `kotlinx-serialization-json`. DTO `@Serializable` для транзакции и raw (все поля: deleted_at, edited_at, category, currency, raw_id), время → ISO-8601. `SupabaseRemoteSyncSource` — реальный upsert через Postgrest `onConflict="user_id,client_id"`, user_id из сессии. `SyncWorker` — не настроен→success, нет сессии→retry, иначе выгрузка `synced=0` (включая удалённые)→markSynced. `AuthRepository` + UI: карточка «Облачный бэкап» и `AuthDialog` (вход/регистрация email+пароль).
**Решения сам:** `raw_id` в облаке — это локальный client_id связанного raw (без FK на удалённый id, чтобы не резолвить remote-id); схема обновлена. Restore с сервера НЕ делал (не было в задаче). `kotlinx-serialization-json` версия 1.7.1.
**⚠️ Риск (проверить при сборке):** точные имена API supabase-kt 2.6.0 (`upsert{onConflict}`, `signInWith(Email){email/password}`, `sessionStatus`/`SessionStatus.Authenticated`) и **персистентность сессии Auth на Android** (может потребовать прокинуть context) — первые кандидаты на правки, если компиляция/рантайм ругнётся.

## 14. Сессия 23.06.2026 — дедуп повторной доставки пушей

Проблема: `onNotificationPosted` может сработать несколько раз для ОДНОГО уведомления (двойной callback, реконнект листенера при включении доступа, перепост активных уведомлений после ребута) → дублирующие raw + транзакции → испорченный остаток и лимит.

Решение: сигнатура повторной доставки `dedupKey = "${sbn.key}|${postTime}"` (одно уведомление = стабильные key+postTime). Разные покупки имеют разный postTime/key → не режутся (легитимные одинаковые траты НЕ теряются).

- **Схема БД v6→v7:** `raw_notifications.dedup_key TEXT` (nullable) + индекс `index_raw_notifications_dedup_key`. `MIGRATION_6_7` (ALTER ADD COLUMN + CREATE INDEX). `schemas/7.json` сгенерён — createSql и индекс точно совпадают с DDL миграции (на устройстве не упадёт).
- **Дедуп в репозитории** (`FinanceRepositoryImpl.ingestNotification`): проверка `countByDedupKey` + вставка raw в одной `db.withTransaction` → Room сериализует транзакции, два конкурентных пуша не пройдут проверку оба. Дубль → ничего не пишем (ни raw, ни tx, ни сдвиг остатка). `RawNotificationDao.countByDedupKey`.
- **Листенер** строит `dedupKey` и передаёт в `ingestNotification`. Ручной ввод/голос/restore — `dedupKey = null` (не дедупятся).
- Бэкап/sync-DTO не тронуты: мапперы позиционные до `synced`, новое поле с дефолтом `null`.

**Ограничение:** если Kaspi *редактирует* существующее уведомление (тот же key, НОВЫЙ postTime) — это будет принято как новая запись. На практике Kaspi шлёт отдельный пуш на операцию; редактирование не наблюдалось. Если всплывёт — сменить сигнатуру на key без postTime (с окном по времени).

**НЕ проверено на устройстве:** реальное срабатывание дедупа на дублях Kaspi и миграция v7 на живой БД (нет девайса в среде). `assembleDebug` + `testDebugUnitTest` — `BUILD SUCCESSFUL`, тесты зелёные (CLI).
